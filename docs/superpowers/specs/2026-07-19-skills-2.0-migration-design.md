## Objetivo

Migrar los skills propios (first-party) de ABSOLUTE-CLAUDE desde archivos SKILL.md en markdown plano a un formato "Skills 2.0": cada skill lleva una configuración YAML estructurada y un JSON Schema junto a su SKILL.md. Simultáneamente, podar los skills que quedan obsoletos con la próxima arquitectura basada en RAG (subproyectos B–D) y consolidar los skills de diseño solapados en un único skill de UI.

## Formato Skills 2.0

Cada skill vive en `skills/<name>/`:

```
skills/<name>/
├── SKILL.md      # unchanged role: frontmatter (name, description) for Claude Code
│                 # discovery + prose body; gains one closing line pointing at sidecars
├── skill.yaml    # structured config
└── schema.json   # JSON Schema draft 2020-12
```

Claude Code solo impone el frontmatter de SKILL.md; los sidecars son una convención del repo que el modelo lee al invocar el skill y que las herramientas futuras (indexado RAG en el subproyecto B, validadores, generadores) consumen como artefactos legibles por máquina. Es un contrato flexible (soft contract) deliberado.

### Campos de `skill.yaml`

```yaml
name: <kebab-case>        # MUST equal the directory name and SKILL.md frontmatter name
version: 2.0.0
kind: knowledge | tool    # tool = ships executable scripts
triggers: [ ... ]         # activation phrases (extracted from the current description)
commands: [ ... ]         # slash commands exposed, if any
scripts: [ ... ]          # bundled executables, if any
dependencies: [ ... ]     # formalizes former [[wiki-link]] cross-references
schema: ./schema.json
```

### Forma de `schema.json`

`definitions` de nivel superior con dos entradas:

- `inputs` — parámetros tipados que acepta el skill. Skills de conocimiento: string `context` libre y opcional. Skills de herramienta: flags tipados reales.

- `outputs` — forma esperada del resultado, descrita de forma declarativa (p. ej. "subject del commit ≤ 50 caracteres, prefijo de tipo Conventional Commits 1.0.0").

### Adición a SKILL.md

Se añade una línea después del cuerpo del frontmatter: `Config: skill.yaml · Schema: schema.json`, para que el modelo cargue los sidecars al invocar el skill.

## Cambios en el instalador

- `install.sh`: el orden de componentes pasa a ser `rtk → caveman → figma → ui-ux → dev-skills`; se actualizan las listas de ids de `--only`/`--skip`; se actualiza el resumen de verificación.

- `bin/components/ui-ux.sh`: elimina el `git clone` + `git pull --ff-only` del upstream; copia el fork propio como cualquier otro skill. El registro del MCP Magic y las deps npm (framer-motion, gsap) no cambian.
- `uninstall.sh`: conserva las entradas de eliminación de repo-map, dcp-lite, solid, design-patterns, architecture-patterns (limpieza legada); añade `architecture-principles`.

## Validador

`skills/validate-skills.mjs` — Node sin dependencias (convención del repo: solo builtins `node:`):

- Para cada directorio bajo `skills/`: existe SKILL.md y su frontmatter parsea; `name` coincide con el nombre del directorio y con `skill.yaml.name`; `skill.yaml` parsea con los campos requeridos (name, version, kind, triggers, schema); `schema.json` es JSON válido que contiene `definitions.inputs` y `definitions.outputs`; cada entrada de `scripts` existe en disco.
- Sale con código 1 y un listado de errores ante cualquier fallo; sale con 0 en caso contrario.
- Ejecutable de forma independiente y desde el instalador (`install.sh --verify` o como parte del paso de resumen).
